## AGENTS.md को छोटा और स्किमेबल रखें {#small-agents-md}

`AGENTS.md` को ओरिएंटेशन के रूप में लोड किया गया है। यह सबसे छोटी चीज़ होनी चाहिए जो एजेंट को सही ढंग से कार्य करने दे, सब कुछ skills में गहराई से धकेल दिया जाए। इन अनुभागों को लक्ष्य बनाएं और कुछ अन्य:

- **उद्देश्य पंक्ति** - ऐप क्या है और प्राथमिक वर्कफ़्लो पर एक वाक्य।
- **मुख्य नियम** - मुट्ठी भर अपरिवर्तनीय जो हमेशा धारण करने चाहिए (SQL में डेटा, संचालन actions के माध्यम से जाते हैं, AI एजेंट चैट के माध्यम से जाता है, स्कीमा परिवर्तन योगात्मक होते हैं)। छोटी, अनिवार्य गोलियाँ।
- **एप्लिकेशन-स्टेट कुंजी** - `navigation`/चयन/फोकस कुंजी जिसे एजेंट यह जानने के लिए पढ़ता है कि उपयोगकर्ता उनके आकार के साथ क्या देख रहा है।
- **कार्रवाई तालिका** - उद्देश्य के अनुसार कार्रवाई के नाम की एक संक्षिप्त तालिका।
- **Skills सूचकांक** - मौजूद skills की एक सूची और प्रत्येक को कब पढ़ना है।

यदि कोई अनुभाग स्क्रीन से आगे बढ़ता है, तो यह एक कौशल से संबंधित है। `AGENTS.md` उत्तर देता है "यह ऐप क्या है और मैं क्या कर सकता हूं," न कि "मैं वास्तव में कठिन कार्य कैसे करूं।"

```markdown
# Projects App

One workspace for projects, tasks, and notes. Agent and UI share the same SQL
data and the same actions.

## Core Rules

- Data lives in SQL via Drizzle. Use actions for all writes.
- All AI work goes through the agent chat; never call an LLM inline.
- Schema changes are additive only.

## Application State

- `navigation.view`: `home` | `project`
- `navigation.projectId`: selected project on a project page

## Actions

| Action           | Purpose                     |
| ---------------- | --------------------------- |
| `list-projects`  | List accessible projects    |
| `create-project` | Create a project            |
| `update-project` | Rename or archive a project |

## Skills

- `project-imports` — read before importing legacy CSV exports.
- `sharing` — read before exposing a project to other users.
```

## एकल-स्रोत AGENTS.md {#single-source}

एक विहित निर्देश फ़ाइल रखें: `AGENTS.md`. यदि कोई ग्राहक `CLAUDE.md` की अपेक्षा करता है, तो इसे दूसरी प्रति के बजाय `AGENTS.md` का सिम्लिंक बनाएं। हाथ से रखी गई दो फ़ाइलें भटक जाती हैं, और एजेंट विरोधाभासी नियमों के साथ समाप्त हो जाता है। सत्य का एक स्रोत, जहां आवश्यक हो वहां जोड़ा गया।

## SKILL.md फ्रंटमैटर को कहना चाहिए कि AND कब क्या है {#skill-frontmatter}

`description` ही एकमात्र ऐसी चीज है जिसे एजेंट यह तय करते समय देखता है कि किसी कौशल को पढ़ना है या नहीं। इसे दो प्रश्नों का उत्तर देना होगा: कौशल में क्या शामिल है, और इसे कब ट्रिगर करना है। ऐसा विवरण जो केवल विषय का वर्णन करता है, सक्रिय नहीं होगा।

```markdown
---
name: project-imports
description: >-
  How to import projects from the legacy CSV export. Use when the user uploads
  a project CSV or asks to migrate projects from the old system.
---
```

- क्षमता के साथ नेतृत्व करें, फिर एक स्पष्ट **"उपयोग करें जब..."** खंड जोड़ें।
- थोड़ा धक्का-मुक्की करें - अति-ट्रिगरिंग एक ऐसे कौशल को मात देती है जो कभी लोड नहीं होता।
- इसे ~40 शब्दों से कम रखें; इसे प्रत्येक वार्तालाप के संदर्भ में लोड किया जाता है।

## प्रगतिशील प्रकटीकरण {#progressive-disclosure}

`SKILL.md` को सरल, जानने योग्य परत के रूप में लिखें: नियम, इसे कैसे करें, क्या करें/क्या न करें की सूची, और संकेत। लंबे उदाहरण, संपूर्ण फ़ील्ड संदर्भ, API विचित्रताएं और एज-केस तालिकाओं को `references/` फ़ाइलों में डालें, एजेंट केवल तभी पढ़ता है जब उसे उनकी आवश्यकता होती है।

```text
.agents/skills/project-imports/
├── SKILL.md            # rule + happy path + do/don't
└── references/
    └── csv-format.md   # full column spec, encodings, edge cases
```

यह हमेशा भरी हुई सतह को छोटा रखता है और संदर्भ को फूलाए बिना गहराई को मापता है। संपूर्ण कौशल प्रारूप के लिए [Skills Guide](/docs/skills-guide) देखें।

## क्रिया-उन्मुख तालिकाएँ लिखें {#action-tables}

एजेंट गद्य की तुलना में तालिकाओं को तेजी से स्कैन करता है। प्रत्येक ऑपरेशन का वर्णन करने वाले पैराग्राफ के बजाय उद्देश्य के लिए नाम की तालिका को प्राथमिकता दें। यही बात राज्य कुंजियों, फ़ील्ड प्रकारों और किसी भी गणना योग्य सेट पर लागू होती है। जब आप कोई क्रिया जोड़ते हैं तो तालिकाएँ स्किम करने योग्य, अलग करने योग्य और सिंक में रखने में आसान होती हैं।

## स्पष्ट टूल विवरण लिखें {#tool-descriptions}

कार्रवाई विवरण उपकरण विवरण हैं - वे उपकरण चयन को संचालित करते हैं। प्रत्येक को एक सटीक, एकल-उद्देश्यीय वाक्य बनाएं:

- यह बताएं कि यह क्या करता है और क्या लौटाता है, यह नहीं कि इसे कैसे कार्यान्वित किया जाता है।
- इसके `.describe()` में प्रत्येक पैरामीटर का वर्णन करें ताकि एजेंट इसे सही ढंग से भर सके।
- प्रति कार्य एक जिम्मेदारी। यदि विवरण में "और भी..." की आवश्यकता है, तो इसे विभाजित करें।
- केवल पढ़ने के लिए actions (`readOnly: true` या `http: { method: "GET" }`) को चिह्नित करें ताकि एजेंट को पता चले कि वे स्वतंत्र रूप से कॉल करने के लिए सुरक्षित हैं।

```ts
defineAction({
  description: "Create a project. Returns the new project id and title.",
  schema: z.object({
    title: z.string().min(1).describe("Project title shown in the sidebar"),
  }),
  // ...
});
```

## Skills बनाम actions {#skills-vs-actions}

Skills और actions पूरक हैं। एक कौशल वह मार्गदर्शन है जिसे एजेंट पढ़ता है; एक
कार्रवाई वह कोड है जिसे एजेंट चला सकता है।

| आवश्यकता                                                                     | उपयोग करें                     |
| ---------------------------------------------------------------------------- | ------------------------------ |
| एजेंट को वर्कफ़्लो, नीति, चेकलिस्ट या रूब्रिक का पालन करना होगा              | **कौशल**                       |
| एजेंट को उदाहरण, संदर्भ सामग्री, या डोमेन-विशिष्ट नियमों की आवश्यकता होती है | **कौशल**                       |
| एजेंट को ऐप डेटा पढ़ने या लिखने की जरूरत है                                  | **कार्रवाई**                   |
| एजेंट को बाहरी API को कॉल करने या अनुमोदन करने की आवश्यकता है                | **कार्रवाई**                   |
| एजेंट सही ऑपरेशन को कॉल करता है लेकिन गलत तरीके से                           | **कौशल** में सुधार करें        |
| एजेंट विश्वसनीय रूप से ऑपरेशन शुरू नहीं कर सकता                              | **कार्रवाई** में सुधार करें    |
| एजेंट गलत टूल चुनता है                                                       | **कार्य विवरण** में सुधार करें |

अधिकांश वास्तविक सुविधाएँ दोनों का उपयोग करती हैं: कौशल बताता है कि कार्य को कैसे पूरा करना है, और
क्रिया टाइप किया गया ऑपरेशन प्रदान करती है। उदाहरण के लिए, एक `invoice-review` कौशल
समीक्षा नीति और वृद्धि नियमों की व्याख्या कर सकता है, जबकि `list-invoices`,
`flag-invoice`, और `approve-invoice` actions वास्तविक पढ़ते और लिखते हैं।

## एंटी-फैब्रिकेशन में बेक करें और पहले सत्यापित करें {#anti-fabrication}

ऐप निर्देशों में ईमानदारी और सत्यापन को डिफ़ॉल्ट व्यवहार बनाना चाहिए:

- **कभी भी मनगढ़ंत बातें न करें।** यदि डेटा नहीं मिलता है या कोई कार्रवाई विफल हो जाती है, तो ऐसा कहें और पुनर्प्राप्त करें - परिणाम का आविष्कार न करें या सफलता का दावा न करें। रिपोर्ट करने से पहले किसी क्रिया या क्वेरी के माध्यम से वास्तविक मूल्य पढ़ें।
- **पूरा घोषित करने से पहले सत्यापित करें।** परिवर्तन के बाद, यह मानने के बजाय कि लिखावट काम कर गई है, इसे रीड-बैक (पंक्ति को फिर से क्वेरी करें, स्क्रीन को `view-screen` के माध्यम से दोबारा पढ़ें) के साथ इसकी पुष्टि करें।
- **पुनर्प्राप्त करें, हार न मानें।** पुनर्प्राप्ति योग्य त्रुटि (एक विफल क्वेरी, एक क्षणिक फ़ेच) पर, कार्य को छोड़ने के बजाय पुनः प्रयास करें या इनपुट को ठीक करें। इसे निर्माण-रोधी नियम से अलग रखें - "बातें न बनाएं" को "पहली त्रुटि पर रुकें" के साथ न मिलाएं।

इन्हें `AGENTS.md` में मुख्य नियमों के रूप में रखें ताकि वे हर मोड़ पर लागू हों।

## एजेंट जिन चार सतहों को देखता है {#four-surfaces}

आपके लेखक द्वारा दिया गया प्रत्येक मार्गदर्शन चार सतहों में से एक में उतरता है। यह जानना कि किस सतह का उपयोग करना है, दोहराव और गलत विवरण को रोकता है:

| सतह                        | इसे कौन लिखता है        | जब यह लोड हो जाता है                                 | वहां क्या है                                                                   |
| -------------------------- | ----------------------- | ---------------------------------------------------- | ------------------------------------------------------------------------------ |
| `AGENTS.md` निर्देश        | आप (डेवलपर)             | प्रत्येक मोड़, अभिविन्यास के रूप में                 | उद्देश्य, मुख्य नियम, राज्य कुंजी, क्रिया सूचकांक, skills सूचकांक              |
| Skills (`SKILL.md`)        | आप (डेवलपर)             | मांग पर जब एजेंट निर्णय लेता है कि कौशल प्रासंगिक है | किसी विशिष्ट पैटर्न के लिए चरण-दर-चरण कैसे करें, क्या करें/क्या न करें की सूची |
| कार्रवाई विवरण (उपकरण)     | आप (डेवलपर)             | हर मोड़, टूल सूची के रूप में                         | क्रिया क्या करती है, यह क्या लौटाती है, पैरामीटर शब्दार्थ                      |
| `application_state` संदर्भ | आपका UI कोड (रनटाइम पर) | प्रत्येक मोड़, लाइव ऐप स्थिति के रूप में             | वर्तमान नेविगेशन, चयन, केंद्रित वस्तु, URL                                     |

**त्वरित निदान:**

- "एजेंट यह पूछता रहता है कि कौन सा रिकॉर्ड खुला होने पर भी उस पर कार्रवाई करनी है" → ठीक करें: अपने UI से वर्तमान आइटम आईडी को `application_state` (`navigation` कुंजी) में लिखें। यह एक `application_state` अंतर है, कौशल अंतर नहीं।
- "एजेंट गलत कार्रवाई करता है या पैरामीटर का दुरुपयोग करता है" → ठीक करें: पैरामीटर पर कार्रवाई के `description` और `.describe()` में सुधार करें। यह एक टूल विवरण फिक्स है, कोई कौशल नहीं।

## क्या कहां जाता है {#what-goes-where}

- **AGENTS.md** - पूरे ऐप पर लागू होता है, हर मोड़ पर: उद्देश्य, मुख्य नियम, राज्य कुंजी, एक्शन इंडेक्स, skills इंडेक्स।
- **Skills** - एक विशिष्ट पैटर्न के लिए पुन: प्रयोज्य, मांग पर लोड किया गया। ऐप में काम करने वाले सभी लोगों पर लागू होता है।
- **मेमोरी (`memory/MEMORY.md`)** — प्रति-उपयोगकर्ता प्राथमिकताएं और सुधार, लिखित मार्गदर्शन नहीं।
