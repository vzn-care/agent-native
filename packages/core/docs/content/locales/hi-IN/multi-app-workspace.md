# मल्टी-ऐप वर्कस्पेस

> **कौन सा कार्यक्षेत्र दस्तावेज़?** यह पृष्ठ **परिनियोजन आकार** को कवर करता है - एक मोनोरेपो, कई ऐप्स, साझा प्रमाणीकरण और एक एकीकृत तैनाती। कार्यस्थान क्या है (अनुकूलन परत: `AGENTS.md`, `LEARNINGS.md`, व्यक्तिगत मेमोरी, skills, कस्टम एजेंट) के लिए [Workspace](/docs/workspace) देखें; शासन के लिए (कौन समीक्षा करता है, अनुमोदन करता है और उसका स्वामी है) [Workspace Governance](/docs/workspace-management) देखें।

जब किसी आंतरिक टूल को वाइब-कोड करने में एक दोपहर लग जाती है, तो आप एक बार भी नहीं रुकते। एक टीम के पास एक CRM, एक सपोर्ट इनबॉक्स, एक डैशबोर्ड, एक ऑप्स कंसोल - दस छोटे ऐप्स, प्रत्येक स्वतंत्र रूप से तैयार किए गए होते हैं। यह तब तक बहुत अच्छा है जब तक आपको उन सभी में कुछ बदलने की आवश्यकता न हो।

उस समय प्रत्येक ऐप का अपना `AGENTS.md`, अपना स्वयं का ऑथ प्लगइन, अपना स्वयं का कॉपी-पेस्ट लेआउट घटक, अपना स्वयं का हार्ड-कोडित Slack टोकन, एक "संगठन" क्या है इसका अपना विचार होता है। एक अनुपालन नियम परिवर्तन का अर्थ है दस पीआर। एक API कुंजी को घुमाने का अर्थ है दस पुनः तैनाती। एक ब्रांड रिफ्रेश का मतलब है दस अलग-अलग हेडर का सिंक से बाहर हो जाना। जिस चीज़ ने उन्हें बनाना आसान बना दिया था, वही चीज़ अब उन्हें प्रबंधित करना कठिन बना रही है।

**मल्टी-ऐप वर्कस्पेस** पैटर्न यह बताता है कि एजेंट-नेटिव इसे कैसे हल करता है। आप अपने सभी ऐप्स को एक निजी `packages/shared` पैकेज के साथ एक मोनोरेपो में होस्ट करते हैं। फ़्रेमवर्क सामान्य डिफ़ॉल्ट का स्वामी है; `packages/shared` केवल कोड, निर्देश, skills, घटकों, या प्लगइन ओवरराइड के लिए है जो वास्तव में आपके कार्यक्षेत्र के लिए कस्टम हैं। प्रत्येक ऐप मुट्ठी भर स्क्रीन और actions तक सिमट जाता है जो इसे अद्वितीय बनाते हैं।

## क्या साझा किया जाता है {#what-gets-shared}

आपके संगठन के प्रत्येक ऐप को जिस बात पर सहमत होना चाहिए वह `packages/shared` में रह सकती है:

| साझा की गई चीज़           | यह कहाँ रहता है                                                                                            |
| ------------------------- | ---------------------------------------------------------------------------------------------------------- |
| प्रामाणिक / SSO ओवरराइड   | `src/server/index.ts` से `authPlugin` निर्यात करें                                                         |
| संगठन / RBAC नियम         | बेहतर प्रामाणिक संगठन, वैकल्पिक रूप से उस `authPlugin` द्वारा लिपटे हुए                                    |
| एजेंट चैट ओवरराइड         | `src/server/index.ts` से `agentChatPlugin` निर्यात करें                                                    |
| एंटरप्राइज़ एजेंट निर्देश | `AGENTS.md`                                                                                                |
| एजेंट skills              | `.agents/skills/<skill-name>/SKILL.md`                                                                     |
| साझा एजेंट actions        | `actions/*.ts`                                                                                             |
| साझा React घटक            | `src/client/index.ts` से निर्यात करें                                                                      |
| डिज़ाइन टोकन / ब्रांड     | एक साझा CSS फ़ाइल जोड़ें और इसे प्रत्येक ऐप से आयात करें                                                   |
| साझा API क्रेडेंशियल      | फ़्रेमवर्क स्कोप्ड क्रेडेंशियल्स को प्राथमिकता दें; यदि आपको नेमस्पेसिंग की आवश्यकता है तो ही सहायक जोड़ें |

प्रत्येक व्यक्तिगत ऐप _सिर्फ स्क्रीन का एक सेट_ बन जाता है - मार्ग, डैशबोर्ड, दृश्य, डोमेन-विशिष्ट actions। जब तक आप वास्तविक कार्यस्थान अनुकूलन नहीं जोड़ते, तब तक फ़्रेमवर्क डिफ़ॉल्ट बाकी को कवर करता है।

वही सीमा तब लागू होती है जब आपका ऐप किसी अन्य प्रथम-पक्ष ऐप का उपयोग करना चाहता है। एक नया कार्यक्षेत्र डैशबोर्ड जिसे ईमेल, कैलेंडर, एनालिटिक्स और कंपनी-मेमोरी संदर्भ की आवश्यकता होती है, उसे मौजूदा मेल, कैलेंडर, एनालिटिक्स और ब्रेन ऐप्स को लिंक या A2A पर कनेक्टेड पड़ोसियों के रूप में उपयोग करना चाहिए। इसे उन टेम्पलेट्स को क्लोन नहीं करना चाहिए, एक रैपर ऐप नहीं बनाना चाहिए जो उन्हें नेस्ट करता है, या केवल उनके डेटा या एजेंटों तक पहुंच प्राप्त करने के लिए चाइल्ड ऐप्स को अपने अंदर नहीं रखना चाहिए। किसी कॉपी को तभी फोर्क या स्कैफोल्ड करें जब आप स्पष्ट रूप से उस ऐप को कस्टमाइज़ करना चाहते हों।

## आरंभ करना {#getting-started}

कार्यस्थान एजेंट-मूल प्रोजेक्ट का डिफ़ॉल्ट आकार है। मचान एक के साथ:

```bash
npx @agent-native/core@latest create my-company-platform
```

CLI प्रत्येक प्रथम-पक्ष टेम्पलेट का एक बहु-चयन पिकर दिखाता है। आप जितने चाहें उतने चुनें - मेल + कैलेंडर + फॉर्म, उदाहरण के लिए - और वे सभी एक ही कार्यक्षेत्र में प्रमाणीकरण और डेटाबेस डिफ़ॉल्ट साझा करने के लिए तैयार हो जाते हैं।

आपको निजी साझा पैकेज के साथ एक pnpm मोनोरेपो मिलता है, एक रूट `package.json` जो कार्यक्षेत्र खोज को तार-तार करता है, एक साझा `.env`, और आपके द्वारा चुने गए प्रत्येक ऐप के लिए एक उप-निर्देशिका:

```an-file-tree title="एक scaffolded workspace"
{
  "entries": [
    { "path": "package.json", "note": "agent-native.workspaceCore घोषित करता है" },
    { "path": "pnpm-workspace.yaml", "note": "packages: [\"packages/*\", \"apps/*\"]" },
    { "path": ".env.example", "note": "shared env vars: ANTHROPIC_API_KEY, A2A_SECRET, DATABASE_URL, ..." },
    { "path": "packages/shared/", "note": "@my-company-platform/shared" },
    { "path": "packages/shared/src/server/", "note": "plugin overrides केवल ज़रूरत होने पर" },
    { "path": "packages/shared/src/client/", "note": "shared React code केवल ज़रूरत होने पर" },
    { "path": "packages/shared/AGENTS.md", "note": "workspace-wide निर्देश" },
    { "path": "apps/mail/" },
    { "path": "apps/calendar/" },
    { "path": "apps/forms/" }
  ]
}
```

फिर इसे बूट करें:

```bash
cd my-company-platform
cp .env.example .env             # fill in ANTHROPIC_API_KEY, BETTER_AUTH_SECRET, ...
pnpm install
pnpm dev                         # opens Dispatch; other apps start on first visit
```

प्रत्येक ऐप पहले से ही जानता है कि कैसे लॉग इन करना है, समान डेटाबेस साझा करना है और कार्यक्षेत्र `AGENTS.md` को लोड करना है। आपने इसमें से कुछ भी तार-तार नहीं किया - फ्रेमवर्क ने रूट `package.json` में `agent-native.workspaceCore` फ़ील्ड के माध्यम से साझा पैकेज को स्वतः खोजा:

```json
{
  "name": "my-company-platform",
  "agent-native": {
    "workspaceCore": "@my-company-platform/shared"
  }
}
```

## एक और ऐप जोड़ना {#adding-a-new-app}

कार्यस्थान के अंदर कहीं से भी:

```bash
npx @agent-native/core@latest add-app
```

CLI आपके द्वारा पहले से इंस्टॉल किए गए ऐप्स को फ़िल्टर करके फिर से टेम्पलेट पिकर दिखाता है। एक या अधिक चुनें और वे `apps/` के अंतर्गत आ जाएंगे। गैर-संवादात्मक संस्करण:

```bash
npx @agent-native/core@latest add-app crm --template content
```

कोई भी प्रथम-पक्ष टेम्पलेट वर्कस्पेस ऐप के रूप में काम करता है - CLI टेम्पलेट पर एक छोटा **workspacify** ट्रांसफॉर्म चलाता है जो साझा पैकेज को एक डिप के रूप में जोड़ता है और `workspace:*` संदर्भों को हल करता है। बनाए रखने के लिए कोई समानांतर "वर्कस्पेस-ऐप" मचान नहीं।

```bash
pnpm install                     # at the workspace root
pnpm dev
```

बस इतना ही। नए ऐप में हर दूसरे ऐप की तरह ही लॉगिन और वर्कस्पेस निर्देश हैं। साझा ब्रांड, actions, या क्रेडेंशियल केवल तभी जोड़ें जब कार्यक्षेत्र को वास्तव में उनकी आवश्यकता हो।

## आप कहां क्या ओवरराइड करते हैं {#layering}

कार्यस्थान के अंदर एजेंट-मूल ऐप्स इस क्रम में तीन स्थानों से क्रॉस-कटिंग व्यवहार को हल करते हैं:

1. **ऐप स्थानीय** - `apps/<name>/` के अंदर फ़ाइलें (सर्वोच्च प्राथमिकता)
2. **कार्यस्थान साझा** - `packages/shared/` (साझा मध्य परत) के अंदर फ़ाइलें
3. **फ्रेमवर्क डिफ़ॉल्ट** — `@agent-native/core` (न्यूनतम)

मर्ज फ़ाइल नाम से होता है। यदि कोई ऐप एक स्थानीय फ़ाइल प्रदान करता है जो अपस्ट्रीम में भी मौजूद है, तो स्थानीय जीतता है। यदि ऐसा नहीं होता है, तो कार्यस्थान साझा संस्करण लागू होता है। यदि साझा एक भी प्रदान नहीं करता है, तो फ्रेमवर्क डिफ़ॉल्ट शुरू हो जाता है। यह प्लगइन्स, skills, actions, और `AGENTS.md` पर लागू होता है।

जब एक ऐप को कुछ अलग चाहिए, तो एक स्थानीय फ़ाइल छोड़ें:

| ओवरराइड करने योग्य बात | ऐप के अंदर बनाने के लिए फ़ाइल                       |
| ---------------------- | --------------------------------------------------- |
| प्रामाणिक प्लगइन       | `apps/<name>/server/plugins/auth.ts`                |
| एजेंट-चैट प्लगइन       | `apps/<name>/server/plugins/agent-chat.ts`          |
| एक विशिष्ट कौशल        | `apps/<name>/.agents/skills/<skill-name>/SKILL.md`  |
| एक विशिष्ट कार्रवाई    | `apps/<name>/actions/<action-name>.ts`              |
| अतिरिक्त एजेंट निर्देश | `apps/<name>/AGENTS.md` (कार्यस्थान एक के साथ विलय) |

कोई वायरिंग नहीं, कोई कॉन्फ़िगरेशन नहीं। फ़ाइल बनाएं और यह उसे संभाल लेती है।

## साझा व्यवहार संपादित करना {#editing-shared-behavior}

आपके द्वारा कस्टमाइज़ की गई सभी क्रॉस-कटिंग `packages/shared/` में रहती है। `src/server/index.ts` से एक `authPlugin` निर्यात करें और प्रत्येक ऐप इसे अगले डेव रीलोड पर उठाता है। `.agents/skills/` के अंतर्गत एक कौशल जोड़ें और प्रत्येक ऐप का एजेंट इसे देखेगा। `actions/` में एक क्रिया जोड़ें और प्रत्येक ऐप का एजेंट इसे कॉल कर सकता है।

क्योंकि साझा पैकेज एक `workspace:*` निर्भरता है, pnpm इसे प्रत्येक ऐप के `node_modules/` में सिम्लिंक करता है। आप इसे कभी भी बनाते या प्रकाशित नहीं करते हैं - ऐप्स को निर्माण के समय इसमें से जो कुछ भी चाहिए उसे बंडल कर देते हैं।

## रनटाइम वैश्विक संसाधन {#runtime-global-resources}

रेपो के साथ भेजे जाने वाले कोड-स्तरीय डिफ़ॉल्ट के लिए `packages/shared` का उपयोग करें: प्लगइन्स, साझा actions, साझा React कोड, फ़ाइल सिस्टम `AGENTS.md`, और फ़ाइल सिस्टम skills। रनटाइम-संपादन योग्य वैश्विक संदर्भ के लिए डिस्पैच वर्कस्पेस संसाधनों का उपयोग करें जिन्हें व्यवस्थापक बिना कोड परिवर्तन के प्रबंधित करना चाहते हैं।

डिस्पैच संसाधनों का दायरा **सभी ऐप्स** है (प्रत्येक ऐप उन्हें रनटाइम पर प्राप्त करता है, कोई प्रतिलिपि या सिंक चरण नहीं) या **चयनित ऐप्स** (ऐप-विशिष्ट संदर्भ के लिए प्रति ऐप दिया गया है)। संपूर्ण संसाधन-मॉडल तालिका, पथ सम्मेलनों और अनुशंसित स्टार्टर पैक के लिए [Workspace](/docs/workspace#global-resources) देखें।

## प्रमाणीकरण और RBAC {#auth-and-rbac}

प्रत्येक एजेंट-नेटिव ऐप पहले से ही [Better Auth](/docs/authentication) प्लस फ्रेमवर्क की अंतर्निहित संगठन प्रणाली के साथ आता है। किसी कार्यक्षेत्र में, आपको वह प्रत्येक ऐप में निःशुल्क मिलता है, जो समान डेटाबेस द्वारा समर्थित होता है। संपूर्ण बहु-किरायेदारी मॉडल के लिए - संगठन, भूमिकाएँ, डेटा अलगाव - [Multi-Tenancy](/docs/multi-tenancy) देखें।

एंटरप्राइज़-विशिष्ट नियमों (अनुमति-सूची डोमेन, SSO प्रवर्तन, अतिरिक्त भूमिका जांच) के लिए, `packages/shared/src/server/index.ts` से एक `authPlugin` निर्यात करें। कार्यक्षेत्र में प्रत्येक ऐप अब उन नियमों को लागू करता है।

सक्रिय संगठन स्वचालित रूप से प्रवाहित होता है: `session.orgId` → `AGENT_ORG_ID` → SQL पंक्ति स्कोपिंग, इसलिए `org_id` के साथ टैग किया गया डेटा अन्य संगठनों, यहां तक कि एजेंट के लिए भी अदृश्य है। संपूर्ण मॉडल के लिए [Security & Data Scoping](/docs/security) देखें।

## साझा MCP सर्वर {#shared-mcp}

वरीयता के क्रम में कार्यस्थान ऐप्स पर MCP सर्वर साझा करने के लिए अनुशंसित विकल्प:

1. **डिस्पैच वर्कस्पेस MCP संसाधन** - **सभी ऐप्स** स्कोप पर डिस्पैच में `mcp-servers/<name>.json` संसाधन जोड़ें। कार्यक्षेत्र में प्रत्येक ऐप को रनटाइम पर MCP सर्वर प्राप्त होता है, जिसमें कोई फ़ाइल संपादन या पुन: तैनाती नहीं होती है। चयनित ऐप्स को केवल तभी अनुदान दें जब सर्वर ऐप-विशिष्ट हो। टोकन डिस्पैच वॉल्ट में रहते हैं; उन्हें `${keys.NAME}` के साथ संसाधन JSON से संदर्भित करें।

2. **रूट `mcp.config.json`** - वर्कस्पेस रूट पर एक फ़ाइल छोड़ें और वर्कस्पेस में प्रत्येक ऐप समान MCP सर्वर से कनेक्ट हो जाता है। अलग-अलग ऐप्स अपने स्वयं के `mcp.config.json` (ऐप-रूट जीत) के साथ ओवरराइड कर सकते हैं। स्थानीय/फ़ाइल सिस्टम MCP सर्वर (`@modelcontextprotocol/server-filesystem`, `claude-in-chrome`, Playwright) के लिए इसका उपयोग करें जिन्हें प्रति-उपयोगकर्ता वॉल्ट क्रेडेंशियल की आवश्यकता नहीं है।

3. **सेटिंग्स UI (व्यक्तिगत/संगठन क्षेत्र)** - दूरस्थ HTTP MCP सर्वर के लिए, उपयोगकर्ता उन्हें व्यक्तिगत या टीम (संगठन) दायरे में सेटिंग्स UI से जोड़ सकते हैं - कोई फ़ाइल संपादन नहीं, रनिंग एजेंट में हॉट-रीलोड किया गया।

कॉन्फ़िगरेशन स्कीमा, प्राथमिकता नियम और हब सेटअप के लिए [MCP Clients](/docs/mcp-clients) देखें।

## साझा पर्यावरण चर {#shared-env}

वर्कस्पेस रूट `.env` स्वचालित रूप से प्रत्येक ऐप में लोड होता है। साझा कुंजियों को एक बार रूट पर रखें - `ANTHROPIC_API_KEY`, `A2A_SECRET`, `BETTER_AUTH_SECRET`, `DATABASE_URL`, `BUILDER_PRIVATE_KEY`, आदि - और हर ऐप उन्हें चुन लेता है। प्रति-ऐप ओवरराइड `apps/<name>/.env` में जाते हैं और संघर्ष पर जीत हासिल करते हैं।

रनटाइम ऐप क्रेडेंशियल के लिए, `.env` फ़ाइलों को हाथ से संपादित करने के बजाय डिस्पैच वॉल्ट को प्राथमिकता दें। वॉल्ट डिफ़ॉल्ट रूप से सभी ऐप्स तक पहुंच प्रदान करता है, इसलिए प्रत्येक सहेजी गई वॉल्ट कुंजी प्रत्येक कार्यस्थान ऐप के लिए उपलब्ध है और इसे `sync-vault-to-app` के साथ पुश किया जा सकता है। वॉल्ट को मैन्युअल मोड में तभी स्विच करें जब ऐप्स को स्पष्ट प्रति-कुंजी अनुदान की आवश्यकता हो।

```text
my-company-platform/
├── .env                           # shared: ANTHROPIC_API_KEY=... , A2A_SECRET=... , ...
└── apps/
    └── mail/
        └── .env                   # optional overrides just for mail
```

कुछ ऑनबोर्डिंग प्रवाह बॉक्स से बाहर कार्यक्षेत्र-जागरूक हैं:

- **Builder `/cli-auth`**: किसी भी ऐप से "कनेक्ट Builder" पर क्लिक करने से `BUILDER_PRIVATE_KEY` और मित्र **वर्कस्पेस रूट** `.env` पर लिखते हैं, इसलिए प्रत्येक ऐप एक ही बार में ब्राउज़र एक्सेस प्राप्त कर लेता है।
- **Env-vars सेटिंग्स रूट** (`POST /_agent-native/env-vars`): जब किसी कार्यक्षेत्र के अंदर, डिफ़ॉल्ट रूप से कार्यक्षेत्र रूट `.env` लिखा जाता है। एक ऐप को ओवरराइड करने के लिए बॉडी में `scope: "app"` पास करें।

## साझा क्रेडेंशियल {#shared-credentials}

एक ही कार्यक्षेत्र में ऐप्स डिफ़ॉल्ट रूप से एक ही `DATABASE_URL` पर इंगित करते हैं, इसलिए फ्रेमवर्क क्रेडेंशियल स्टोरेज प्रत्येक ऐप के लिए प्रति-ऐप कॉन्फिगरेशन के बिना एक क्रेडेंशियल उपलब्ध करा सकता है। यदि आपका कार्यक्षेत्र सख्त नामकरण परंपरा चाहता है तो सीधे `@agent-native/core/credentials` का उपयोग करें, या `packages/shared` में एक पतला सहायक जोड़ें।

## साझा डिज़ाइन टोकन {#design-tokens}

फ्रेमवर्क Tailwind v4 पर है। एक साझा CSS फ़ाइल को `packages/shared` में तभी जोड़ें जब कार्यक्षेत्र में साझा करने के लिए वास्तविक ब्रांड टोकन हों, फिर इसे प्रत्येक ऐप के `app/global.css` से आयात करें:

```css
@import "tailwindcss";
@import "@my-company-platform/shared/styles/tokens.css";
@source "./**/*.{ts,tsx}";

:root {
  --background: 0 0% 100%; /* ...brand tokens... */
}
.dark {
  --background: 220 6% 6%; /* ... */
}
```

ब्रांड रंग, टाइपोग्राफी, रिक्ति स्केल, और कोई भी साझा घटक वर्ग उस एक CSS फ़ाइल में रह सकते हैं। इसे `packages/shared` में अपडेट करें और प्रत्येक ऐप अगले बिल्ड पर रीब्रांड हो जाएगा।

## तैनाती {#deployment}

आपके पास दो विकल्प हैं: **एकीकृत तैनाती** (कार्यस्थानों के लिए डिफ़ॉल्ट) या प्रति-ऐप स्वतंत्र तैनाती।

### एकीकृत परिनियोजन (अनुशंसित)

एक कमांड कार्यक्षेत्र में प्रत्येक ऐप बनाता है और उन्हें एक ही मूल, प्रति ऐप एक पथ के पीछे भेजता है:

```bash
npx @agent-native/core@latest deploy
# https://your-agents.com/mail/*       → apps/mail
# https://your-agents.com/calendar/*   → apps/calendar
# https://your-agents.com/forms/*      → apps/forms
```

प्रत्येक ऐप `APP_BASE_PATH=/<name>` और `VITE_APP_BASE_PATH=/<name>` के साथ बनाया गया है और चयनित Nitro प्रीसेट के माध्यम से उत्सर्जित होता है। क्लाउडफ्लेयर पेज डिफ़ॉल्ट प्रीसेट है और `dist/_worker.js` प्लस `_routes.json` पर एक डिस्पैचर वर्कर का उपयोग करता है। Netlify `npx @agent-native/core@latest deploy --preset netlify` के साथ समर्थित है; यह `.netlify/functions-internal/<app>-server` के तहत ऐप फ़ंक्शंस उत्सर्जित करता है और रीडायरेक्ट उत्पन्न करता है जो स्थैतिक संपत्तियों को अप्रचलित छोड़ देता है इसलिए CDN पहले फ़ाइलों को प्रस्तुत करता है। वर्सेल `npx @agent-native/core@latest deploy --preset vercel` के साथ समर्थित है; यह वर्सेल के बिल्ड आउटपुट API का उपयोग करके एक रूट `.vercel/output` बंडल लिखता है।

**समान मूल** पर होना ही वास्तविक लाभ है:

- **साझा लॉगिन सत्र।** बेटर ऑथ अपनी कुकी को शीर्ष डोमेन पर सेट करता है, इसलिए किसी भी ऐप में लॉग इन करने से आप हर ऐप में लॉग इन हो जाते हैं। कोई क्रॉस-डोमेन SSO नृत्य नहीं।
- **ज़ीरो-कॉन्फिग क्रॉस-ऐप A2A।** `@mail` टैगिंग `@calendar` एक समान-मूल फ़ेच बन जाता है - कोई CORS नहीं, कोई JWT भाई-बहनों के बीच हस्ताक्षर नहीं। बाहरी A2A आज भी JWT का उपयोग करता है।
- **एक DNS रिकॉर्ड, एक प्रमाणपत्र, एक CDN कैश।**

`dist/` आउटपुट प्रकाशित करें:

```bash
wrangler pages deploy dist
```

नेटलिफ़ाई के लिए:

```bash
npx @agent-native/core@latest deploy --preset netlify --build-only
```

वर्सेल गिट परिनियोजन के लिए, बिल्ड कमांड को यहां सेट करें:

```bash
npx @agent-native/core@latest deploy --preset vercel --build-only
```

### सार्वजनिक ऐप रूट

वर्कस्पेस ऐप्स डिफ़ॉल्ट रूप से आंतरिक होते हैं। केवल-लॉगिन व्यवस्थापक पृष्ठों वाली सार्वजनिक साइट के लिए, एक सार्वजनिक दर्शक सेट करें और उस ऐप के `package.json` में व्यवस्थापक उपसर्ग को सुरक्षित रखें:

```json
{
  "agent-native": {
    "workspaceApp": {
      "audience": "public",
      "protectedPaths": ["/admin"]
    }
  }
}
```

कुछ सार्वजनिक पृष्ठों वाले अधिकतर आंतरिक ऐप्स के लिए, ऑडियंस को आंतरिक और सूची पृष्ठ उपसर्गों को छोड़ दें:

```json
{
  "agent-native": {
    "workspaceApp": {
      "publicPaths": ["/", "/share"]
    }
  }
}
```

ये सेटिंग्स केवल रीड-ओनली पेज नेविगेशन को प्रभावित करती हैं। फ़्रेमवर्क टूल, एजेंट चैट, A2A, वॉल्ट एक्सेस और मनमाना API तब तक प्रमाणित रहते हैं जब तक कि ऐप स्पष्ट रूप से `createAuthPlugin({ publicPaths: [...] })` के साथ सार्वजनिक उपसर्गों की घोषणा नहीं करता।

### प्रति-ऐप स्वतंत्र तैनाती

प्रत्येक ऐप को उसके अपने डोमेन (`mail.company.com`, `calendar.company.com`) पर प्राथमिकता दें? कार्यक्षेत्र में प्रत्येक ऐप अभी भी एक स्वतंत्र तैनाती योग्य है - `cd apps/mail && npx @agent-native/core@latest build` बिल्कुल एक स्टैंडअलोन मचान की तरह व्यवहार करता है। क्रॉस-ऐप A2A फिर साझा `A2A_SECRET` के साथ मानक JWT-हस्ताक्षरित पथ से गुजरता है। अलग-अलग तैनात ऐप्स के बीच क्रॉस-डोमेन SSO को हब के रूप में डिस्पैच के साथ पहचान महासंघ द्वारा नियंत्रित किया जाता है - [Cross-App SSO](/docs/cross-app-sso) देखें; एकीकृत एकल-मूल परिनियोजन इसकी आवश्यकता से बचाता है।

### साझा डेटाबेस, साझा क्रेडेंशियल

आप जो भी चुनें, बॉक्स के बाहर क्रॉस-ऐप स्थिति के लिए प्रत्येक ऐप को एक ही `DATABASE_URL` पर इंगित करें: उपयोगकर्ता खातों का एक सेट, संगठनों का एक सेट, साझा सेटिंग्स का एक सेट। यदि प्रत्येक ऐप का अपना डेटाबेस है, तो कार्यक्षेत्र पैटर्न अभी भी काम करता है - आप बस उस साझा-स्थिति की कहानी को खो देते हैं।

साझा पैकेज स्वयं कभी भी स्टैंडअलोन तैनात नहीं किया जाता है। यह एक `workspace:*` डिपो है जो pnpm प्रत्येक ऐप के `node_modules/` में सिमलिंक करता है, इसलिए प्रत्येक ऐप पारदर्शी रूप से बिल्ड समय पर जो कुछ भी चाहिए उसे बंडल करता है।

## दायरे से बाहर (अभी के लिए) {#out-of-scope}

कार्यस्थान पैटर्न जानबूझकर संकीर्ण है। कुछ चीज़ें जो यह जानबूझकर अभी तक नहीं संभाल पाया है:

- **एन्क्रिप्टेड क्रेडेंशियल वॉल्ट।** रनटाइम ऐप क्रेडेंशियल के लिए डिस्पैच वॉल्ट को प्राथमिकता दें ([Shared environment variables](#shared-env) देखें)। गैर-वॉल्ट फ़ॉलबैक पथ - फ़्रेमवर्क `settings` तालिका में सीधे लिखे गए साझा क्रेडेंशियल - उन्हें आज सादे पाठ के रूप में संग्रहीत करता है, इसलिए जब आप इस पर भरोसा करते हैं तो जिम्मेदारी से घुमाएँ।
- **साझा कोड को निजी npm पर प्रकाशित करना।** साझा पैकेज केवल `workspace:*` है; निजी रजिस्ट्री के माध्यम से मल्टी-रेपो साझाकरण संभव है, लेकिन इसे सीमित नहीं किया जा सकता है।
- **विचारित घटक लाइब्रेरी।** `packages/shared` वह जगह है जहां _आप_ साझा घटक डालते हैं। फ़्रेमवर्क shadcn/ui या किसी अन्य सिस्टम को उस स्लॉट में बाध्य नहीं करता है।
