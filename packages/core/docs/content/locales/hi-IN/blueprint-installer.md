# ब्लूप्रिंट इंस्टालर

> **यह किसके लिए है:** मेजबान लेखक और एक प्रदाता, चैनल जोड़ने वाले इंटीग्रेटर्स
> सैंडबॉक्स बैकएंड, या किसी रेसिपी को उनके कोडिंग एजेंट में डालकर रेपो पर कार्रवाई।

`agent-native add` एक गूंगा स्कैफ़ोल्डर नहीं है जो आपके लिए फ़ाइलें लिखता है। यह स्टडआउट करने के लिए एक क्यूरेटेड Markdown _इंटीग्रेशन ब्लूप्रिंट_ उत्सर्जित करता है। आप उस ब्लूप्रिंट को अपने स्वयं के कोडिंग एजेंट (Claude कोड, Codex,…) में पाइप करते हैं, जो पूरे संदर्भ के साथ लाइव रेपो के विरुद्ध परिवर्तनों को लागू करता है।

यह एजेंट-लागू-परिवर्तन, फ़ाइल सिस्टम-फर्स्ट हाउस शैली में फिट बैठता है: फ्रेमवर्क नुस्खा प्रदान करता है (स्पर्श करने के लिए कैनोनिकल फ़ाइलें, सम्मान करने के नियम, सत्यापन चरण), और कोडिंग एजेंट संपादन करता है।

```bash
agent-native add provider stripe | claude
agent-native add channel discord  | codex
```

## प्रयोग {#usage}

```bash
agent-native add <kind> <name>            # print a curated blueprint
agent-native add <kind> <https://docs…>   # research-and-integrate from a URL
agent-native add --list                   # list available kinds and blueprints
```

- एक खाली **नाम** `blueprints/<kind>/<name>.md` से एक क्यूरेटेड ब्लूप्रिंट का समाधान करता है।
- ए **URL** एक नाम के बजाय उस तरह के लिए एक सामान्य _शोध-और-एकीकृत_ ब्लूप्रिंट उत्सर्जित करता है, जिसमें URL अनुसंधान प्रारंभिक बिंदु के रूप में एम्बेडेड होता है (URL एक शोध बीज है, एक ज्ञात नुस्खा नहीं)।
- ब्लूप्रिंट **stdout** पर जाता है; डायग्नोस्टिक्स stderr पर जाता है, इसलिए `… | claude` केवल ब्लूप्रिंट प्राप्त करता है।

## सीडेड ब्लूप्रिंट {#seeded}

`agent-native add --list` दिखाता है कि बॉक्स में क्या भेजा गया है:

| प्रकार     | नाम       | यह क्या सेट करता है                                                                |
| ---------- | --------- | ---------------------------------------------------------------------------------- |
| `provider` | `stripe`  | एक प्रदाता को `provider-api` सब्सट्रेट (कैटलॉग/दस्तावेज़/अनुरोध तिकड़ी) से जोड़ें। |
| `channel`  | `discord` | एक `PlatformAdapter` इनबाउंड वेबहुक चैनल लागू करें और इसे पंजीकृत करें।            |
| `sandbox`  | `docker`  | डॉकर कंटेनर में `run-code` चलाने के लिए `SandboxAdapter` सीम लागू करें।            |
| `action`   | `crud`    | Zod स्कीमा (N के ऊपर एक `update`) के साथ एक एकल बहु-सतह `defineAction` जोड़ें।     |

प्रत्येक ब्लूप्रिंट स्व-निहित है: इसे पढ़ने वाले कोडिंग एजेंट को छूने के लिए फ़ाइलें मिलती हैं, फ्रेमवर्क नियमों का सम्मान किया जाता है (actions सत्य का एकल स्रोत है, कभी भी हार्डकोड रहस्य नहीं, स्वामित्व योग्य डेटा का दायरा, `packages/*` स्रोत के लिए एक परिवर्तन जोड़ें), और एक ठोस **सत्यापित करें** अनुभाग।

## URL → अनुसंधान ब्लूप्रिंट {#url}

जब आप URL पास करते हैं तो इस प्रकार के लिए कोई क्यूरेटेड नुस्खा नहीं होता है (या एक नया एकीकरण चाहते हैं), `add` बीज के रूप में URL के साथ एक सामान्य "शोध-और-एकीकृत" ब्लूप्रिंट उत्सर्जित करता है:

```bash
agent-native add provider https://docs.example.com/api | claude
```

जेनरेट किया गया ब्लूप्रिंट कोडिंग एजेंट को वास्तविक एंडपॉइंट, ऑथ मॉडल, पेलोड आकार और हस्ताक्षर/सत्यापन आवश्यकताओं के लिए URL (और इसे लिंक करने वाले पेज) लाने के लिए कहता है - प्रशिक्षण डेटा से अनुमान लगाने के लिए _नहीं_ - फिर लागू करें और सत्यापित करें। इसमें प्रकार-विशिष्ट मार्गदर्शन भी होता है (उदाहरण के लिए एक `provider` URL को `provider-api` सब्सट्रेट की ओर ले जाया जाता है; एक `channel` URL को `PlatformAdapter` की ओर ले जाया जाता है)।

## अपना स्वयं का खाका जोड़ना {#authoring}

Markdown फ़ाइल को `packages/core/blueprints/<kind>/<name>.md` में छोड़ें। प्रकार उपनिर्देशिका है; नाम `.md` के बिना फ़ाइल नाम है। इसे स्वचालित रूप से उठाया जाता है - `--list`, नाम रिज़ॉल्यूशन, और कैटलॉग सभी निर्देशिका को रनटाइम पर पढ़ते हैं। इसे पंजीकृत करने के लिए किसी कोड परिवर्तन की आवश्यकता नहीं है।

ब्लूप्रिंट `.md` फ़ाइलें `package.json` `files` में `blueprints` प्रविष्टि के माध्यम से प्रकाशित पैकेज में भेजी जाती हैं, इसलिए वे अंतिम उपयोगकर्ताओं के लिए `node_modules/@agent-native/core/blueprints/**` पर हल होती हैं।

प्रत्येक ब्लूप्रिंट को बिना किसी अन्य संदर्भ के कोडिंग एजेंट के लिए एक निर्देश सेट के रूप में लिखें। एक अच्छे ब्लूप्रिंट में:

1. **एक-पंक्ति लक्ष्य** और "आप एक एजेंट-नेटिव ऐप में एक कोडिंग एजेंट हैं, इन्हें वास्तविक स्रोत परिवर्तन के रूप में लागू करें" फ़्रेमिंग।
2. **पहले पढ़ें** - सटीक फ़ाइलें जो अनुबंध हैं।
3. **स्पर्श करने योग्य फ़ाइलें** — ठोस पथ और प्रत्येक परिवर्तन क्या करता है।
4. **फ्रेमवर्क नियमों का सम्मान** - actions-प्रथम, कोई हार्डकोडेड रहस्य नहीं, स्वामित्व योग्य डेटा का दायरा, प्रकाशन योग्य-पैकेज स्रोत के लिए एक परिवर्तन जोड़ें।
5. **सत्यापित करें** - टाइपचेक, एक केंद्रित `*.spec.ts`, और एक एंड-टू-एंड चेक।

> [!TIP]
> मौजूदा प्रकार के तहत एक नए क्यूरेटेड ब्लूप्रिंट को किसी कोड की आवश्यकता नहीं है - लेकिन यदि आप एक बिल्कुल नई तरह की निर्देशिका बनाते हैं, तो वह प्रकार स्वचालित रूप से `--list` में भी दिखाई देता है।
